## Executive conclusion

Epic does not expose one unified secret-domain framework. It composes independent legacy mechanisms: a context record masked from a root-derived value and slate identifier, random aggregate-signature nonces, hierarchical output and address derivation, a password-encrypted seed file, separate API-secret files, optional encrypted API bodies, and transport-specific keys. Following Epic directly would inherit undocumented cross-domain assumptions, legacy formats, and parameters. DOM V3 may retain only protected properties and implementation lessons; every V3 cryptographic construction still requires independent DOM-specific approval.

DEC-V3-SECRET-DOMAINS remains BLOCKING, owned by CRYPTOGRAPHIC_REVIEW with HIGH severity. No decision status, specification, protocol rule, derivation formula, domain label, KDF profile, nonce algorithm, scalar mapping, or authenticated-encryption construction is selected by this study.

## Private transaction context and lifecycle

Epic `Context` holds a parent derivation identifier, private aggregate-signature key, private aggregate-signature nonce, input and output identifiers with amounts, fee, participant position, and optional payment-proof derivation index (`libwallet/src/types.rs:620-666`, `Context`; EXPLICIT_EPIC_CONSTRUCTION). The backend indexes it by slate identifier bytes plus participant position (`impls/src/backends/lmdb.rs:348-378`, `get_private_context`; `854-897`, `save_private_context` and `delete_private_context`).

The stored record serializes as a JSON-backed database value (`impls/src/serialization.rs:23-47`, `Serializable`; `impls/src/backends/db.rs:178-277`, `Batch::put`). Before storage, only the two scalar fields are XOR-masked with values derived from a root key, the slate identifier, and two fixed purpose strings; retrieval reverses that operation (`impls/src/backends/lmdb.rs:63-94`, `private_ctx_xor_keys`; `854-879`). This is an explicit Epic construction, not authenticated storage encryption. The inspected database code contains JSON serialization and SQLite write batching, but no record-level authenticated encryption, associated data, replay defense, or rollback witness for contexts (`impls/src/backends/db.rs:43-70`, `178-277`; EXPLICIT_EPIC_CONSTRUCTION plus EPIC_EVIDENCE_INSUFFICIENT for integrity and rollback resistance).

Send, invoice, and invoice-processing paths save context before returning the slate (`libwallet/src/api_impl/owner.rs:457-463`, `514-520`, `603-609`). Locking and finalization retrieve the same keyed context; successful finalization deletes it in a subsequent batch (`libwallet/src/api_impl/owner.rs:631-659`; `libwallet/src/api_impl/foreign.rs:163-184`). Thus a restart can recover an existing stored context if the keychain and mask are available; a crash between separate context and transaction-state batches can leave durable context without a demonstrated single atomic lifecycle unit. Retry reuses the stored context when lookup succeeds, but no complete pinned-tree test establishes context non-reuse across distinct slates, participants, crashes, or deletion failures.

Classification: private context is EXPLICIT_EPIC_CONSTRUCTION for structure, indexing, persistence, and deletion; IMPLICIT_SEPARATION_BY_CALL_PATH_OR_STORAGE for lifecycle ordering; and EPIC_EVIDENCE_INSUFFICIENT for authenticated persistence, complete crash atomicity, rollback resistance, and uniqueness proof. DOM V3 may adopt the protected property of durable context-before-later-signing and retained restart evidence, but requires DOM-specific replacement and independent cryptographic approval.

## Nonces, blindings, derivation, and allocation

`Context::new` obtains the private signature nonce through the aggregate-signature nonce generator in normal operation (`libwallet/src/types.rs:644-667`). Transaction construction derives the context private key from the transaction's blinding contribution and supplies the nonce into round-one and round-two signing (`libwallet/src/internal/selection.rs:223-258`; `libwallet/src/internal/tx.rs:150-200`, `203-273`; `libwallet/src/slate.rs:299-378`). Kernel offset generation explicitly uses random scalar generation and transforms the context signing key by the offset (`libwallet/src/slate.rs:501-532`). Participant data publishes the corresponding public excess and public nonce beside a participant position (`libwallet/src/slate.rs:67-104`, `436-488`).

These paths show nonce and blinding separation by field and call path, but do not define a V3-style global purpose framework. The complete pinned-tree searches found no direct test named for nonce reuse, blinding reuse, private-context reuse, or cross-participant reuse. The available transaction and repost tests exercise end-to-end transaction and stored-transaction flows (`controller/tests/transaction.rs:35-240`; `controller/tests/repost.rs:111-184`), not a cryptographic uniqueness oracle.

Output allocation advances and persists a per-parent child index before returning an identifier (`impls/src/backends/lmdb.rs:453-488`, `next_child`). Send and receive builders use that allocator for fresh output and change material (`libwallet/src/internal/keys.rs:21-33`; `libwallet/src/internal/selection.rs:223-290`, `488-516`). Account paths are derived and persisted independently (`libwallet/src/internal/keys.rs:62-104`; `impls/src/backends/lmdb.rs:159-164`). Recovery scanning raises a recovered parent child index to at least the observed maximum plus one (`libwallet/src/internal/scan.rs:512-546`). These are protected monotonic-allocation properties, but not a proof that all random signing material is non-reusable after every durable or exposed boundary.

Classification: transaction nonce and blinding are EXPLICIT_EPIC_CONSTRUCTION for generation and signing use, with IMPLICIT_SEPARATION_BY_CALL_PATH_OR_STORAGE for roles; output, change, receive, coinbase, and account allocation are LEGACY_COMPATIBILITY_BEHAVIOR; global non-reuse is EPIC_EVIDENCE_INSUFFICIENT. DOM V3 may retain monotonic allocation and durable-before-exposure properties, but must reject direct adoption of Epic scalar, offset, nonce, slate, and derivation mechanisms.

## Seed, password, backend, and recovery evidence

Epic stores a mnemonic-derived seed in a JSON seed file and derives a keychain with a network boolean (`impls/src/lifecycle/seed.rs:31-84`, `WalletSeed`; `impls/src/lifecycle/default.rs:216-243`, `open_wallet`). The seed envelope uses a fresh salt and nonce, a PBKDF2-HMAC-SHA512 password derivation, and ChaCha20-Poly1305 with empty associated data (`impls/src/lifecycle/seed.rs:214-309`, `EncryptedWalletSeed`). It parses hex fields and returns an encryption error for malformed ciphertext, wrong password, or invalid nonce; its inline test checks round trip, wrong password, and malformed nonce (`impls/src/lifecycle/seed.rs:313-335`). Password change backs up, deletes, recreates, rereads, compares, then optionally removes the backup (`impls/src/lifecycle/default.rs:299-353`). Seed recovery recreates the seed file from the mnemonic; scan restores output observations and raises child indexes, but does not reconstruct private contexts or transaction history (`impls/src/lifecycle/seed.rs:119-143`; `libwallet/src/internal/scan.rs:447-560`).

The wallet backend holds serialized records in SQLite and stores finalized transactions separately as hexadecimal serialized files (`impls/src/backends/lmdb.rs:403-430`; `impls/src/backends/db.rs:43-70`, `178-277`). Only context scalar fields receive the XOR transformation already described. The inspected code provides no full-backup envelope, no demonstrated encrypted stored-transaction format, no authenticated database record envelope, and no cross-device rollback prevention. These are NO_EPIC_EQUIVALENT_FOUND or EPIC_EVIDENCE_INSUFFICIENT, not evidence that a safe DOM V3 construction exists.

Classification: seed protection is EXPLICIT_EPIC_CONSTRUCTION; backend context masking is EXPLICIT_EPIC_CONSTRUCTION but LEGACY_COMPATIBILITY_BEHAVIOR; backup and recovery are PROTECTED_PROPERTY_ONLY for seed recovery and monotonic scan reconstruction, with NO_EPIC_EQUIVALENT_FOUND for a V3 full-backup secret-domain envelope. Every Epic cryptographic parameter and format requires DOM-specific replacement and independent approval.

## Authentication, chain, purpose, and encoding evidence

Epic creates separate owner and node API secret files containing random alphanumeric strings (`config/src/config.rs:34-43`, `76-126`), while a secure owner API token contains an optional keychain mask (`api/src/types.rs:31-38`; `api/src/owner_rpc_s.rs:2487-2491`). API-body encryption uses a random nonce with AES-256-GCM and empty associated data (`api/src/types.rs:58-149`); an API round-trip test exists (`api/src/types.rs:314-350`). Epic message transport separately derives a shared secret and encrypts a message envelope with a different PBKDF2 and ChaCha20-Poly1305 composition (`libwallet/src/message.rs:29-149`). Address and payment-proof material follows a separate account-path-derived key path and conversion route (`libwallet/src/address.rs:31-64`; `libwallet/src/api_impl/owner.rs:437-455`). Tor and Epicbox paths introduce further distinct transport keys (`impls/src/tor/config.rs:80-120`, `170-215`; `impls/src/adapters/epicbox.rs:136-181`).

Network selection is present when deriving the seed keychain and configuration paths (`impls/src/lifecycle/seed.rs:72-75`; `config/src/config.rs:45-57`, `174-194`). The context mask binds to slate identifier and separates its two scalar fields with fixed strings (`impls/src/backends/lmdb.rs:63-94`), and the context key also includes participant position (`impls/src/backends/lmdb.rs:348-358`). The inspected evidence does not establish a binding of every secret construction to chain ID, wallet identity, transaction identity beyond slate keying, participant identity beyond storage position, record type, backup version, or a common purpose registry. Slate public fields use explicit serializers and version fields (`libwallet/src/slate.rs:67-87`, `164-218`; `libwallet/src/slate_versions/ser.rs:15-47`), but backend JSON serialization and the empty-AAD envelopes do not provide a V3 canonical secret-envelope policy.

Classification: authentication and transport separation are IMPLICIT_SEPARATION_BY_CALL_PATH_OR_STORAGE plus LEGACY_COMPATIBILITY_BEHAVIOR; chain binding is EPIC_EVIDENCE_INSUFFICIENT; purpose binding is EXPLICIT_EPIC_CONSTRUCTION only for the context-mask inputs and otherwise EPIC_EVIDENCE_INSUFFICIENT; canonical encoding is EXPLICIT_EPIC_CONSTRUCTION for selected public fields but EPIC_EVIDENCE_INSUFFICIENT for V3 secret envelopes. DOM V3 must reject direct adoption and obtain independent cryptographic approval.

## DOM implications for the independent reviewer

The reviewer must decide, from DOM authority rather than Epic: one purpose-bound construction or an approved set of constructions for private contexts, backup records, credentials, and transport material; chain, wallet, account, transaction, participant, record-type, and version bindings; canonical encodings; authenticated storage and backup envelopes; KDF and key-expansion policy; nonce, blinding, allocation, exposure, deletion, and restore floors; wrong-chain and rollback behavior; and test vectors, negative cases, properties, and interoperability evidence. The reviewer must also decide whether any legacy DOM compatibility material can be isolated without letting it select new V3 cryptography.

The study appends only evidence to [the cryptographic review request](../docs/FOUNDATIONAL_CRYPTOGRAPHIC_REVIEW_REQUEST.md). No specification or decision status changed. The Epic protected properties suitable for conceptual adoption are durable context-before-later-signing, deterministic child allocation advancement, conservative scan floor repair, explicit failure on malformed seed envelope input, and separation of operational secret roles. Epic-specific mechanisms rejected as direct templates include its context XOR derivation, secret labels, seed and message envelopes, API token and encrypted-body scheme, derivation paths, address conversion, slate format, transport keys, database serialization, and all cryptographic constants.
